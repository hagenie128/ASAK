# ASAK UX Philosophy

> Status: Current

## 1. UX 목표

ASAK의 UX는 사용자가 빠르게 누르는 경험보다 **실수 없이 끝내는 경험**을 목표로 한다.

샐러드 주문은 선택지가 많기 때문에 클릭 수만 줄이면 오히려 오류가 커질 수 있다.

따라서 ASAK는 다음을 우선한다.

- 현재 선택 상태를 명확히 보여주기
- 필수와 선택을 구분하기
- 추가와 제외를 구분하기
- 가격 변화를 바로 보여주기
- 실패 시 이전 선택을 유지하기
- 다음 행동을 한 개로 제한하기

---

## 2. Kiosk UX Rules

### 터치

- 최소 터치 영역 80 × 80px
- 주요 CTA 높이 100~120px
- 작은 텍스트 링크로 핵심 행동을 제공하지 않는다.

### 정보

- 메뉴 카드에는 이미지, 메뉴명, 가격, 칼로리, 품절 여부를 제공한다.
- 옵션 선택은 그룹별로 분리한다.
- 필수 여부와 선택 가능한 개수를 항상 표시한다.
- 장바구니에서는 옵션 전체를 과도하게 펼치지 않고 요약한다.

### 흐름

- 뒤로 가도 선택 상태를 유지한다.
- 결제 실패 시 장바구니를 유지한다.
- 완료 후에는 주문번호와 대기 주문 수를 우선한다.
- 타임아웃 전에 경고와 계속 주문 선택지를 제공한다.

## 3. Admin UX Rules

### Dashboard

로그인 직후 주문 목록 하나만 보여주지 않는다.

관리자는 다음을 동시에 판단해야 한다.

- 오늘 매출
- 주문 수
- 평균 객단가
- 진행 중 주문
- 품절
- 인기 메뉴

따라서 SCR-022 Dashboard를 별도 화면으로 둔다.

### Live Order Board

실시간 조리 업무에 집중한다.

- 접수
- 준비 중
- 완료
- TTS 호출

### Order Management

검색·필터·기간·상세 확인에 집중한다.

실시간 보드와 관리 목록은 목적이 다르므로 같은 화면으로 취급하지 않는다.

## 4. 왜 예상 조리시간 대신 대기 주문 수인가

예상 조리시간은 다음 문제를 가진다.

- 메뉴 구성과 직원 숙련도에 따라 오차가 크다.
- 시스템이 정확한 시간을 계산하기 어렵다.
- 실제보다 늦거나 빠르면 신뢰가 떨어진다.

대기 주문 수는:

- 계산이 단순하다.
- 고객이 현재 혼잡도를 직접 이해할 수 있다.
- 잘못된 시간 약속을 만들지 않는다.

따라서 완료 화면은 `waitingOrderCount`를 사용한다.

---

## 5. 왜 장바구니에 옵션 수정과 삭제를 분리하는가

두 행동은 결과가 완전히 다르다.

- 옵션 수정: 주문 항목 유지
- 삭제: 주문 항목 제거

한 메뉴나 작은 아이콘에 묶으면 실수 가능성이 높다.

따라서 별도 Action으로 제공하고, 삭제는 ConfirmDialog를 사용한다.

---

## 6. 왜 TTS를 사용하는가

관리자 화면에서 완료 상태만 바뀌면 고객은 자신의 주문 완료를 알기 어렵다.

TTS는:

- 별도 호출 장비 없이 브라우저에서 구현 가능하다.
- 직원의 반복 호출 부담을 줄인다.
- 주문 완료 흐름을 실제 매장처럼 보여준다.

단, 서버 상태 변경 성공 후에만 호출하며 실패해도 주문 상태는 유지한다.

---

## 7. 카피 원칙

- 짧다.
- 해결 방법을 포함한다.
- 사용자를 탓하지 않는다.
- 과한 감탄사를 사용하지 않는다.
- 버튼은 행동을 표현한다.

홈 카피:

- 오늘도 아삭아삭
- 당신의 하루를 신선하게

브랜드 톤은 친근하지만 유아적이지 않아야 한다.
