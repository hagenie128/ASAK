# ASAK 프로젝트 전체 흐름 및 로직 가이드

> 이 문서는 프론트엔드와 백엔드가 어떻게 연결되는지, 그리고 어떤 순서로 기능을 구현해야 하는지 한 번에 이해하기 위한 가이드입니다.

---

## 1. 프로젝트를 이해하는 핵심 관점

ASAK는 단순한 화면 구현 프로젝트가 아니라, 아래 흐름을 하나의 주문 시스템으로 연결하는 프로젝트입니다.

1. 고객이 화면에서 메뉴를 고른다.
2. 프론트엔드가 선택 정보를 정리해서 백엔드로 보낸다.
3. 백엔드가 주문 정보를 검증하고 저장한다.
4. 결제/상태 변경이 이루어지고 결과를 다시 화면에 반영한다.

즉, 핵심은 “사용자 행동 → 프론트 상태 관리 → API 요청 → 백엔드 처리 → DB 저장 → 응답 → 화면 반영”입니다.

---

## 2. 전체 시스템 흐름

### 2-1. 최상위 흐름

```text
사용자 행동
  → 프론트 화면 전환 / 상태 변경
  → API 요청
  → 백엔드 검증 / 서비스 로직 처리
  → DB 저장 또는 조회
  → 응답 반환
  → 프론트 화면 갱신
```

### 2-2. 실제 고객 주문 흐름

```text
홈 화면
  → 주문 방식 선택 (먹고가기 / 포장)
  → 메뉴 목록 확인
  → 메뉴 상세 / 옵션 선택
  → 장바구니 담기
  → 주문 확인
  → 결제 요청
  → 주문 완료
```

이 흐름은 프론트와 백엔드가 각각 책임을 나눠서 처리합니다.

---

## 3. 프론트엔드가 하는 일

프론트엔드는 “사용자 경험을 만드는 역할”입니다. 사용자가 클릭하는 모든 행동을 화면 위에서 처리하고, 필요한 정보를 서버에 요청합니다.

### 프론트엔드의 핵심 책임

- 화면을 그린다.
- 사용자의 선택 상태를 관리한다.
- 장바구니 상태를 유지한다.
- API를 호출한다.
- 백엔드 응답을 화면에 반영한다.
- 에러 상황을 사용자에게 보여준다.

### 프론트엔드가 관리하는 상태

1. UI 상태
   - 현재 화면
   - 모달 열림/닫힘
   - 탭 선택 상태
   - 로딩 상태

2. 비즈니스 상태
   - 선택한 메뉴
   - 선택한 옵션
   - 수량
   - 장바구니 총액
   - 주문 정보

### 프론트엔드 로직 예시

예를 들어 메뉴를 선택했을 때의 흐름은 다음과 같습니다.

```text
사용자 클릭
  → 선택된 메뉴 상태 업데이트
  → 옵션 선택 UI 표시
  → 옵션 값 반영
  → 총 가격 재계산
  → 장바구니에 담기 버튼 활성화
  → 장바구니 상태 저장
```

즉, 프론트엔드는 “상태를 바꾸고, 화면을 다시 그리는 역할”입니다.

---

## 4. 백엔드가 하는 일

백엔드는 “데이터와 비즈니스 규칙을 처리하는 역할”입니다. 프론트에서 넘어온 요청이 올바른지 검증하고, 필요한 데이터를 저장하거나 조회합니다.

### 백엔드의 핵심 책임

- 요청 데이터를 받는다.
- 입력값을 검증한다.
- 비즈니스 규칙을 적용한다.
- DB에 저장하거나 조회한다.
- 응답 형식을 통일해서 반환한다.

### 백엔드 로직 예시

주문 생성 요청이 들어오면 보통 아래 순서로 처리합니다.

```text
POST /orders
  → 요청 DTO 검증
  → 메뉴/옵션 정보 확인
  → 주문 총액 계산
  → 주문 엔티티 생성
  → 주문 아이템 저장
  → 결제 상태 초기화
  → 응답 반환
```

즉, 백엔드는 “정책과 데이터 처리”를 담당합니다.

## 5. 프론트와 백엔드의 역할 분담

| 영역             | 프론트엔드 | 백엔드 |
| ---------------- | ---------- | ------ |
| 화면 표시        | O          | X      |
| 사용자 입력 처리 | O          | X      |
| 상태 관리        | O          | X      |
| API 요청         | O          | O      |
| 비즈니스 규칙    | X          | O      |
| DB 저장/조회     | X          | O      |
| 응답 포맷 통일   | X          | O      |

이 분담을 꼭 기억하면 개발 범위를 어디까지 맡아야 하는지 훨씬 명확해집니다.

## 6. 데이터 흐름 기준

ASAK 프로젝트에서는 데이터가 대략 아래 흐름으로 움직입니다.

### 6-1. 초기 데이터 준비

```text
크롤링/시드 데이터 생성
  → JSON/seed 데이터 준비
  → 백엔드에 적재 또는 연동
  → 프론트에서 API로 조회
```

### 6-2. 메뉴 조회 흐름

```text
프론트가 메뉴 목록 요청
  → 백엔드가 DB에서 메뉴/카테고리 조회
  → 응답을 JSON으로 반환
  → 프론트가 카드 형태로 화면에 출력
```

### 6-3. 주문 생성 흐름

```text
프론트가 주문 정보 전송
  → 백엔드가 요청 유효성 검증
  → 주문 생성 및 저장
  → 결제 상태 초기화
  → 주문 번호/주문 ID 반환
  → 프론트가 완료 화면 표시
```

## 7. 구현할 때 가장 먼저 잡아야 할 로직

프로젝트를 시작할 때는 아래 흐름부터 먼저 이해하는 것이 좋습니다.

### 7-1. 메뉴 선택 로직

- 카테고리 선택
- 메뉴 카드 렌더링
- 메뉴 상세 보기
- 옵션 선택
- 가격 재계산

### 7-2. 장바구니 로직

- 메뉴 추가
- 수량 증가/감소
- 옵션 반영
- 총액 계산
- 삭제/수정

### 7-3. 주문 로직

- 주문 정보 수집
- 주문 생성 요청
- 결제 상태 반영
- 주문 완료 표시

### 7-4. 관리자 로직

- 주문 목록 조회
- 주문 상태 변경
- 품절 상태 변경
- 메뉴/결제 설정 관리

## 8. 개발 순서로 보면 좋은 흐름

### Phase 4 — 관리자 기능

- 주문 목록/상태 관리

- 관리자 메뉴 관리

## 9. 개발할 때 기억하면 좋은 규칙

### 프론트엔드 규칙

- 화면은 “흐름” 기준으로 나눈다.
- 상태는 “기능” 기준으로 관리한다.
- API 호출은 분리해서 관리한다.
- 공통 UI는 재사용 가능한 형태로 만든다.

### 백엔드 규칙

- Controller는 얇게 유지한다.
- 핵심 로직은 Service로 분리한다.
- 요청/응답은 DTO로 명확히 나눈다.
- 에러는 통일된 응답 형식으로 반환한다.

### 공통 규칙

- 프론트와 백엔드가 주고받는 데이터 형식은 항상 명확하게 맞춘다.
- 화면 요구사항은 API 명세와 연결되어 있어야 한다.
- 하나의 기능을 구현할 때는 “사용자 행동 → 서버 처리 → 결과 표시” 순서로 생각한다.

---

## 10. 한 문장 요약

ASAK 프로젝트는 “고객이 키오스크에서 메뉴를 고르고 주문하는 흐름”을 프론트엔드 화면, 백엔드 API, DB 저장이 함께 연결된 하나의 시스템으로 구현하는 프로젝트입니다.

즉, 개발할 때는 항상 아래 순서로 생각하면 됩니다.

```text
사용자 행동
→ 프론트 상태 변경
→ API 요청
→ 백엔드 처리
→ DB 반영
→ 응답 반환
→ 화면 갱신
```
