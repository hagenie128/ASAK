# 1. 프로젝트 비전

**한 줄 정의**: ASAK(아삭) 가상 브랜드를 기반으로, 옵션 조합형 키오스크 풀스택 시스템을 설계·시연하여 프랜차이즈 운영 흐름을 학습·포트폴리오로 정리한다. (유사 실제 매장 리서치는 벤치마킹 목적으로 참고)

**왜 이 프로젝트인가**

- 옵션 선택형 메뉴 구조는 백엔드(관계형 데이터 모델링)와 프론트(상태관리·UI) 양쪽 역량을 균형 있게 보여줄 수 있음
- 재고관리 확장 가능성이 있어 '단순 주문 데모'를 넘어서는 서비스형 설계로 발전 가능

# 2. 타겟 사용자 정의

| 페르소나 | 특징 | 키오스크에서 원하는 것 |
| --- | --- | --- |
| 직장인 (평일 점심) | 시간 압박, 반복 방문 | 빠른 재주문, 즐겨찾기 |
| 다이어트/헬스 유저 | 칼로리·영양 정보 민감 | 옵션별 칼로리 표시 |
| 첫 방문 고객 | 메뉴 이해도 낮음 | 직관적 분류, 추천 조합 |

**설계 원칙**: 위 3개 페르소나 중 하나라도 화면에서 헷갈리면 실패로 간주. 특히 '첫 방문 고객' 기준으로 UI 테스트할 것.

# 3. 범위 정의 (MVP vs 확장)

### Week 5 MVP (8/1 목표, SCR-001~008)

- [ ]  카테고리별 메뉴 조회
- [ ]  메뉴 상세 조회
- [ ]  옵션 선택
- [ ]  장바구니 담기/수정/삭제
- [ ]  주문 생성
- [ ]  가상 결제 처리
- [ ]  주문 완료 화면
- [ ]  관리자 주문 목록 조회 및 상태 변경
- [ ]  판매 항목 품절 여부 관리 (옵션기능, 일정 여유 있으면 함께 구현)

### 이후 주차 확장 기능

- [ ]  메뉴 품절 관리 고도화
- [ ]  옵션 단위 품절 처리
- [ ]  칼로리/알레르기 표시 고도화
- [ ]  추천 조합(베스트 메뉴) 하이라이트
- [ ]  관리자용 매출/주문 통계
- [ ]  다국어(영/한) 전환
- [ ]  영수증/QR/바코드 등 장치 연동

### 하지 않을 것 (명확히 제외)

- 실제 결제 연동(PG사 연동) — 결제 완료 화면까지만 시뮬레이션
PG사 연동 → 네이버페이, 카카오페이, 토스페이, 애플페이 같은 **간편결제 서비스**를 의미
- 회원가입/로그인 — 비회원 주문 흐름만 구현
- 다매장/다지점 관리 — 단일 매장 기준

> ⚠️ **Week 5 MVP** 목표는 고객 키오스크 주문 흐름(SCR-001~008) 완성입니다. 전체 프로젝트는 **9주**(7/2~9/2)이며, 관리자·장치·멤버십 등은 Week 6~8 확장으로 관리해야 스코프 크립(scope creep)을 막을 수 있습니다.

# 4. 성공 기준 (발표 평가 기준 역산)

| 항목 | 성공 기준 |
| --- | --- |
| 기능 완성도 | 고객 주문 → 옵션 선택 → 장바구니 → 가상 결제 → 관리자 확인 흐름이 실제 동작 |
| 코드 품질 | API 명세와 실제 구현 일치, 최소한의 예외처리 존재 |
| UX | 첫 방문 가정 사용자가 안내 없이 주문 완료 가능 |
| 발표 | 실기기/데모 환경에서 오류 없이 시연 |
| 협업 흔적 | Git 커밋 로그로 팀원 전원의 기여가 드러남 |

**발표 시연 시나리오 (미리 정해두기)**

1. 신규 고객이 메뉴 둘러보며 옵션 조합 확인
2. 고객이 장바구니에서 수량을 조정하고 가상 결제까지 완료
3. 관리자가 주문 목록에서 접수-준비중-완료 상태를 변경하는 모습

# 5. 리스크 관리

| 리스크 | 발생 가능성 | 대응 방안 |
| --- | --- | --- |
| 프론트-백엔드 API 스펙 어긋남 | 높음 | 2주차에 API 명세 먼저 확정, 이후 변경 시 즉시 공유 |
| 옵션 조합 로직이 생각보다 복잡해짐 | 중간 | MVP 범위에서 조합 규칙 최대한 단순화 (05번 페이지 옵션 구조 규칙표 기준) |
| 팀원 일정 충돌(과제/시험 등) | 중간 | 매주 월요일 이번 주 가용시간 공유, 백로그 우선순위로 조정 |
| 발표 직전 통합 실패 | 높음 | 6주차를 통합 전용 주차로 확보, 7주차에 여유 buffer |
| 재고관리 로직 과설계 | 중간 | "품절 여부 boolean" 수준으로 단순화, 수량 추적은 확장 기능으로 미룸 |

# 6. UX/UI 설계 원칙

- **터치 우선**: 버튼 최소 44x44px, 텍스트 최소 16px
- **핵심 화면 전환 5단계 이내**: 메뉴선택 → 옵션선택 → 장바구니 → 주문확인 → 결제, 그 이상 화면을 늘리지 않기 (FWD-UI-002, 이전의 "3탭"→"4단계" 표현은 실제 화면설계(04번 문서)의 주문확인 단계를 반영하지 못해 2026-07-04 기준 5단계로 재수정. TC-002 기대결과도 함께 수정함)
- **되돌리기 쉬움**: 어느 화면에서든 이전 단계로 쉽게 복귀 가능해야 함
- **오류 상태 명확히**: 품절/네트워크 오류 시 텍스트로 명확히 안내 (아이콘만으로 표현 금지)
- **일정시간 미입력 시 자동 초기화** (FWD-SYS-001): 앞팀(스터디카페 도메인) 사례 참고하여 추가

# 7. 아키텍처 개요

```jsx
[React 프론트] --Axios--> [Spring Boot API] --JPA--> [MySQL]
     |                          |
  장바구니/주문 상태 관리    메뉴/옵션/주문/결제 테이블
```

**핵심 테이블 (05번 페이지에 3차 정규화 기준 22개 테이블로 상세 반영됨, 아래는 핵심 흐름만 요약)**

- category, code_group, common_code (카테고리/공통코드)
- menu, ingredient, allergen, menu_ingredient, ingredient_allergen (메뉴와 재료/알레르기 구조)
- option_group / menu_option_group / menu_option / option_item (옵션 구조)
- orders / order_item / order_item_option / item_exclusion (주문과 선택 옵션/제외 재료)
- payment (가상 결제 내역)

> MVP 필수 17개 + 확장 대비 5개(tag, menu_tag, menu_nutrition, option_item_component, payment_method_config) 구성입니다. 상세는 05번 문서를 기준으로 합니다.
