# ASAK Figma 종합 수정 실행 계획 — 사람 vs AI 역할 분담

**작성일**: 2026-07-17
**근거 문서**: [figma-qa-unified-complete-2026-07-17.md](figma-qa-unified-complete-2026-07-17.md)(출처 A~E 통합) + [ASAK_FIGMA_DESIGN_FEEDBACK_COMPONENT_SCREEN_2026-07-17.md](ASAK_FIGMA_DESIGN_FEEDBACK_COMPONENT_SCREEN_2026-07-17.md)(출처 C)
**목적**: "AI로만 전체 계획 돌리니까 제자리걸음"이라는 문제를 해결 — 사람이 직접 하면 빠른 것과 AI가 해야 정확한 것을 분리해서, 판단이 필요한 곳에서 AI가 왕복하며 멈추는 일을 없앤다.

---

## 0. 왜 지금까지 제자리걸음이었나 (원인 분석)

지금까지의 세션 기록(메모리)을 보면 반복된 패턴이 있다:

1. **여러 세션이 같은 Figma 파일을 동시에 병렬로 작업** — 한 세션이 고친 걸 다른 세션이 모르고 되돌리거나 충돌(예: CartItemCard showEditButton 회귀 사고, D1 Pilot 소유권 충돌).
2. **"어떤 값이 맞는지"를 AI가 계속 재판단** — 출처마다 권장값이 다른데(제목 18px vs 22~24px, 가격색 #6C9700 vs #4A7A00 등) AI가 이걸 스스로 결정하지 못하고 재확인·재QA를 반복. 이건 **AI가 잘하는 일이 아니라 사람이 5초면 고를 수 있는 취향/기준 문제**다.
3. **판단이 필요한 항목과 기계적으로 처리 가능한 항목이 섞여서 하나의 거대한 "전체 계획"으로 취급됨** — 그래서 사람 결정 1개가 막히면 그 뒤 AI 작업 전체가 같이 멈춤.

**이번 계획의 원칙**:
- 사람이 결정할 것은 **먼저, 한 번에, 표로** 받는다(1단계, 5~15분).
- 그 값이 정해지면 이후 작업은 AI가 **재판단 없이 그대로 실행**한다.
- Figma에서 클릭 몇 번이면 끝나는 것은 AI에게 시키지 않는다(설명하는 시간이 더 걸림).
- 여러 컴포넌트·수십 개 인스턴스에 걸친 반복 작업, 전수 검색·대조, 코드 로직은 AI가 한다(사람이 손으로 하면 누락·오타 위험).
- **Figma 문제가 아니라 실제 프론트엔드 코드 문제인 것**(총액 계산 등)은 별도 트랙으로 분리한다 — Figma를 아무리 고쳐도 안 풀리는 항목이었다.
- 한 번에 **한 세션만** Figma에 쓴다(병렬 금지). 완료 후 스크린샷 1회 검증으로 끝내고, 무한 재확인 루프를 만들지 않는다.

## 0.5 출처(A~E) 신뢰도 평가 — 누구 말을 계속 듣고, 누구한테 검증받을지

5개 출처는 같은 파일을 서로 다른 방식으로 조사했다. "어느 출처가 제일 맞다"고 한 줄로 말할 수 없는 이유는, 애초에 **조사한 층위 자체가 다르기 때문**이다. 아래는 각 출처가 실제로 무엇을 잘했고 무엇을 놓쳤는지, 통합 문서(FIGMA_QA_UNIFIED_COMPLETE)와 C 문서 원문에 남은 증거를 근거로 정리한 것 — 통합자(이번 계획 작성자)의 막연한 인상이 아니라 각 출처 스스로 남긴 기록에 근거했다.

| 출처 | 무엇을 조사했나 | 강점(근거) | 약점(근거) |
|---|---|---|---|
| **A**(Cursor 스크린 실측) | 05-C 57개 + 06-C 72개 **화면 프레임** 전수 픽셀 실측 | 화면에서 실제로 벌어지는 문제를 가장 많이 잡음(SCR-008 `940×200`, Dialog 680/440 혼용, CategoryTap이 실제 화면에 쓰이는 것 확인, Toast가 화면에서 299×74로 작게 나오는 것 확인) — **"사용자가 실제로 보는 화면"** 기준으로는 가장 촘촘함 | 초기에 "Menu Detail이 OrderDetailRow×46을 오용한다"고 봤다가 나중에 "실제로는 OptionCard×46 사용 중"으로 스스로 정정(A.0 전역 정정 표) — 화면만 보고 구조를 판단하면 이런 오판이 나올 수 있음을 스스로 보여준 사례 |

SCR-009 총액0원·SCR-018 목데이터 미분기 등 신규 P0도 이 교차검증 과정에서 발견 | Figma API 호출 한도에 두 번 걸려 **관리자 화면 45/72(62.5%)만 실측**, 키오스크 화면 색상 정밀검증도 40개 화면 미완 — 스스로 문서에 이 한계를 명시해뒀지만 커버리지가 100%는 아님 |

| **E**(Figma DS QA) | 시각·터치·행간(line-height) 중심, ✅⚠️🔴 심각도 태깅 | 유일하게 **폰트크기 전체(44~11px)에 대한 lh 매핑표**를 통째로 제공 — 다른 출처는 "행간이 AUTO다"까지만 지적하고 끝나는데 E는 실제 넣을 숫자까지 줌. "즉시수정 7"처럼 실행 우선순위를 짧게 압축한 것도 유용 | 일부 값이 다른 출처보다 유독 더 큰 쪽으로 치우침(터치 72px, radius pill 20, CategoryTab 64) — 근거가 "왜 이 값이어야 하는지"보다 "이게 더 낫다"는 톤이라 다른 출처와 충돌할 때 근거 비교가 상대적으로 어려움 |

### 결론 — 역할 배정

| 질문 | 답 | 이유 |
|---|---|---|
| **판단이 갈릴 때 누구를 우선하나?** | **화면까지 대조한 근거가 있는 쪽**(A 또는 C) **> 마스터만 본 근거**(B) **> 취향 기반 제안**(E) | 사용자가 실제로 보는 게 화면이지 마스터 컴포넌트가 아니다. 다만 A는 구조 오판 전례가 있으니 A 단독 주장도 C로 한 번 더 대조하는 게 안전 |
| **이 실행 계획(3단계 AI 처리)을 계속 진행할 출처는?** | **C(이번 세션, 즉 지금 이 Claude)** | Figma MCP 읽기·쓰기 접근이 이미 인증돼 있고, "마스터만 고치고 끝"이 아니라 화면 재현까지 확인하는 방식이 이번 계획의 핵심 원칙(§0)과 일치. 단, **AI-4(Text Style 연결)·AI-5(day Variant)는 D가 원래 발견한 구조 문제**라 착수 전에 D의 원본 진단 내용을 다시 참고할 것 |
| **수정 완료 후 검증은 누구한테?** | **실행한 사람과 다른 세션/도구**로 — 이상적으로는 **A 방식(화면 전수 스크린샷 재확인)** 또는 **새 C 세션(라이브 재조회)** | B가 스스로 "마스터만 보고 단정했다가 화면 대조로 뒤집힌" 전례를 남겼듯, **고친 사람이 스스로 "잘 고쳤다"고 판단하면 같은 실수가 반복될 위험**이 있다. 다른 세션이 처음부터 다시 화면을 보고 "실제로 바뀌었는지"만 확인하는 게 안전 |
| **E의 제안(터치 72, radius 20 등)은 어떻게 하나?** | 1단계 결정표에서 이미 대부분 "기존 기준 유지"로 정리됨 | 값 자체가 나쁜 게 아니라, 다른 4개 출처와 근거를 맞대봤을 때 "필수"보다 "취향"에 가까운 항목들이라 이번 1차 실행에서는 보류하고 여유 있을 때 다시 검토 |

**한 줄 요약**: 마스터 컴포넌트 이야기는 B, 화면 이야기는 A, 구조 밑바탕 이야기는 D, 행간 숫자는 E, 이 셋을 종합해서 "진짜 급한 것"을 골라낸 게 C — 그래서 실행은 C(이 세션)로 계속 가되, **다 고친 뒤에는 반드시 C가 아닌 새 세션이나 A 방식으로 재확인**해서 "고친 사람만 그렇다고 우기는" 상황을 막는다.

## 1단계 — 사용자가 먼저 결정할 것 (충돌하는 값 중 택1)

아래는 출처 간 권장값이 갈린 항목이다. **이 표를 다 채우기 전에는 AI에게 관련 작업을 시키지 마라** — 어차피 되돌아온다. 각 항목에 추천값을 넣어뒀으니, 그냥 승인만 해도 되고 다른 값을 골라도 된다.

| # | 항목 | 후보 A | 후보 B | 추천 | 이유 |
|---|---|---|---|---|---|
| 1 | ConfirmDialog 제목 크기(키오스크) | 18 Bold(E) | 22~24(B) | **22~24** | 키오스크 최소 본문 16 기준, Modal(22)과 위계 통일 |
| 2 | ConfirmDialog 키오스크 폭 | 560~640(B) | 560~680(A 실측 680 존재) | **680** | 이미 화면에 680 사례가 있어 그쪽으로 통일이 변경 범위 작음 |
| 3 | MenuCard 가격 텍스트색 | `#6C9700`(B·C, 이미 사용중) | `#4A7A00`(E 신규 제안) | **`#6C9700` 유지** | 이미 Foundations Green/600과 정확히 일치하는 기존값, 바꿀 이유 약함 |

| 10 | Toast 키오스크 대응 | size=kiosk variant 신설(B) | 기존 폭 유지(D는 Admin 기준 충분) | **size=kiosk 신설** | 키오스크·관리자 물리적 환경이 다르다는 원칙(§0)과 일치, 억지로 하나로 합치면 한쪽이 항상 손해 |

## 2단계 — 🙋 사용자가 Figma에서 직접 하는 게 더 빠른 것

**공통 이유**: 마스터 컴포넌트의 속성 1~2개만 바꾸면 되고, 값이 이미 정해져 있거나 위 1단계에서 정해지는 것들. Figma에서 노드 선택 → 오른쪽 패널에서 값 변경, 몇 초~몇 분이면 끝난다. AI에게 "이 색을 이 값으로 바꿔줘"라고 설명하고 결과를 검증받는 왕복이 오히려 더 오래 걸린다.

### 🔴 최우선 — 3초짜리인데 가장 크게 보이는 버그

| 항목 | 방법 | 비고 |
|---|---|---|
| **SCR-008 주문완료 BottomCTA `940×200`** | 해당 인스턴스 선택 → 우클릭 → **"Reset to main component"**(변경사항 초기화) | 마스터는 정상(`1080×180`)인데 이 인스턴스만 크기·variant가 override된 것으로 3개 출처(A·B·C)가 동일하게 확인 — override 리셋 한 번으로 끝날 가능성 높음. **AI에게 시키지 말고 지금 바로 Figma에서 확인해볼 것.** |

### 색상 1곳 교체(Master, 이견 없음)

| 항목 | 방법 |
|---|---|
| ConfirmDialog warning primary 버튼 텍스트: 흰색 → 다크(`#0D0D0D`/`#292D30`) | 버튼 텍스트 레이어 선택 → Fill을 Modal과 동일한 다크 토큰으로 교체 |
| EmptyState 버튼 테두리 `#ccc`(raw hex) | 테두리 선택 → 우클릭 "Apply variable" → Semantic Border 토큰 적용 |

## 6. 다음 라운드 — 출처별 "실행 계획" 취합 방법

지금 A~E 5개 출처 각각에게 이것과 같은 "실행 계획" 문서를 만들게 했다면, 5개가 다 나왔을 때 **QA 취합 때와 똑같은 방식(전부 표로 펼쳐서 새 mega-document 만들기)으로 취합하지 말 것**을 권한다. 이유와 대안은 아래와 같다.

### 왜 QA 취합 방식을 그대로 쓰면 안 되나

- QA 취합(`FIGMA_QA_UNIFIED_COMPLETE`)은 "발견을 빠짐없이 모으는 것"이 목적이라 872줄짜리 표가 돼도 괜찮았다. 그런데 그 결과 실행 가능한 형태로 바꾸는 **2차 작업(이 실행계획 문서)이 또 필요했다.**
- "실행 계획" 5개를 다시 그런 식으로 펼쳐서 합치면, 그 결과를 실행 가능하게 만드는 **3차 작업이 또 필요해진다** — 문서가 문서를 낳는 패턴이 반복된다. 지금 `ASAK/docs/design/`에 이미 관련 문서가 20개 이상 쌓여 있는 것 자체가 이 패턴의 증거다.

### 대신 이렇게: 병합(merge)이 아니라 **차이(diff)**만 반영

1. **이 문서(ASAK_FIGMA_FIX_EXECUTION_PLAN)를 기준선으로 고정**한다. 5개를 동등하게 늘어놓고 새로 짜지 않는다.
2. 다른 4개 출처의 실행 계획이 나오면, 이 문서의 **1단계 결정표 / 2단계 체크리스트 / 3단계 Batch 목록**과 항목 단위로 대조해서 셋 중 하나로만 분류한다:
   - **동의** → 아무것도 안 함(재작성 금지)
   - **값·순서가 다름** → 1단계 결정표에 후보값으로 추가(지금 이미 이렇게 돼 있음), 최종 판단은 §0.5 우선순위 원칙(화면 대조 근거 > 마스터만 본 근거 > 취향)으로 정리
   - **이 계획에 없던 새 항목** → "§7 라운드2 반영" 같은 **짧은 추가 섹션**으로만 붙인다. 문서 전체를 다시 쓰지 않는다.
3. 사람/AI 담당 구분이 다른 출처와 다르게 나오면(예: 어떤 출처는 어떤 항목을 "AI가 스크립트로" 라고 봤는데 이 계획은 "사람이 직접"으로 봤다면), 왜 다르게 봤는지 1줄만 남기고 §0.5 기준으로 정리한다.

### 누가 이 diff 작업을 하면 좋은지

- 5개 문서를 동시에 놓고 항목별로 대조하는 건 취향 판단이 아니라 **대조 작업**이라 AI가 하는 게 맞다(3단계 AI 원칙과 동일 — 여러 문서에 걸친 반복 대조는 사람이 손으로 하면 누락 위험).
- 다만 §0.5에서 정한 "실행한 세션이 스스로 검증하지 않는다"는 원칙은 여기도 적용된다 — **이 diff 결과를 최종 확정하는 것만큼은 사람이 5분 정도 훑어보고 "동의" 도장**을 찍는 게 안전하다. AI가 5개 계획을 비교하고 "결론적으로 제 계획이 맞습니다"라고 스스로 결론 내리게 두지 않는다.

### 타이밍

5개가 **다 나온 뒤에 한 번에** diff한다. 하나 나올 때마다 이 문서를 고치면 그 자체가 또 다른 형태의 왕복 루프가 된다.
