# 팀 개발/기획 리소스 가이드

팀프로젝트, 개인 포트폴리오, 실무형 사이드 프로젝트에서 계속 재사용할 수 있도록 개발 코드, 알고리즘, 기획, 디자인, API, DB, 배포, 운영, 발표 자료에 유용한 사이트를 단계별로 정리한 문서입니다.

## 0. 빠른 결론

처음부터 모든 사이트를 다 쓰려고 하면 오히려 산만해집니다. 팀프로젝트 기준으로는 아래 조합만 먼저 잡아도 충분합니다.

| 목적 | 우선 추천 |
| --- | --- |
| 팀 문서/회의록/WBS | Notion |
| 이슈/작업 관리 | GitHub Issues 또는 GitHub Projects |
| 화면 기획/와이어프레임 | Figma, FigJam, Whimsical |
| 디자인 레퍼런스 | Mobbin, Awwwards, Siteinspire, Checklist Design |
| DB 설계 | ERDCloud, [dbdiagram.io](http://dbdiagram.io) |
| API 명세/테스트 | Swagger, Postman |
| 코드 참고 | GitHub Code Search, Stack Overflow, 공식 문서 |
| 알고리즘/코테 | 프로그래머스, 백준, LeetCode |
| 배포 | Vercel, Render, Railway, Netlify |
| 오류/운영 확인 | Sentry, UptimeRobot |
| 발표/포트폴리오 | Behance, Framer, GitHub README |

## 1. 단계별 추천 루트

### 1단계: 아이디어/문제 정의

무엇을 만들지 정하고, 사용자 문제와 핵심 기능을 좁히는 단계입니다.

| 사이트 | 용도 | 팀에서 쓰는 방법 |
| --- | --- | --- |
| [Product Hunt](https://www.producthunt.com/) | 새 서비스/툴 탐색 | 비슷한 서비스가 어떻게 문제를 풀었는지 참고 |
| [AlternativeTo](https://alternativeto.net/) | 대체 서비스 탐색 | 경쟁 서비스와 기능 차이 확인 |
| [G2](https://www.g2.com/) | SaaS 리뷰 확인 | 실제 사용자 불편사항 찾기 |
| [Capterra](https://www.capterra.com/) | 업무용 소프트웨어 비교 | 기능 목록, 가격, 사용자 리뷰 참고 |
| [Reddit](https://www.reddit.com/) | 사용자 의견/불편 탐색 | 실제 사용자의 불만, 니즈, 워크플로우 확인 |
| [Google Trends](https://trends.google.com/) | 검색 트렌드 확인 | 주제의 관심도 변화 확인 |

추천 사용법:

- 비슷한 서비스 3개 찾기
- 공통 기능과 차별 기능 나누기
- 사용자가 불편해하는 지점 3개 정리
- 우리 팀 MVP에서 구현할 기능만 남기기

### 2단계: 요구사항/기획 정리

기능 범위, 사용자 흐름, 화면 목록, 예외 상황을 문서화하는 단계입니다.

| 사이트 | 용도 | 팀에서 쓰는 방법 |
| --- | --- | --- |
| [Notion](https://www.notion.so/) | 문서/회의록/기획서 | 프로젝트 허브, 회의록, WBS, 요구사항 관리 |
| [FigJam](https://www.figma.com/figjam/) | 아이디어 정리 | 브레인스토밍, 사용자 흐름, 역할 분담 |
| [Miro](https://miro.com/) | 화이트보드 협업 | 서비스 구조, 플로우차트, 회의 정리 |
| [Whimsical](https://whimsical.com/) | 플로우차트/와이어프레임 | 화면 흐름, IA, 사용자 시나리오 정리 |
| [Checklist Design](https://www.checklist.design/) | UX 체크리스트 | 로그인, 결제, 폼, 검색 등 화면별 누락 점검 |

기획 단계 체크리스트:

- 사용자는 누구인가?
- 사용자가 가장 먼저 하는 행동은 무엇인가?
- MVP에서 꼭 필요한 기능은 무엇인가?
- 나중에 해도 되는 기능은 무엇인가?
- 실패/예외 상황은 무엇인가?
- 관리자 화면이 필요한가?

### 3단계: 화면 설계/UI 레퍼런스

화면 구조, UX 흐름, 컴포넌트, 디자인 톤을 잡는 단계입니다.

| 사이트 | 용도 | 팀에서 쓰는 방법 |
| --- | --- | --- |
| [Mobbin](https://mobbin.com/) | 실제 앱 UI 패턴 | 로그인, 결제, 마이페이지, 온보딩 참고 |
| [Spotted in Prod](https://www.spottedinprod.com/clips) | iOS 앱 인터랙션 영상 | 마이크로 인터랙션, 전환, 제스처 참고 |
| [Awwwards](https://www.awwwards.com/) | 고급 웹디자인 | 발표용 첫인상, 비주얼 톤 참고 |
| [Siteinspire](https://www.siteinspire.com/) | 큐레이션 웹디자인 | 깔끔한 레이아웃 참고 |
| [Component Gallery](https://component.gallery/) | 컴포넌트 패턴 | 버튼, 탭, 모달, 테이블, 상태값 참고 |
| [Navbar Gallery](https://www.navbar.gallery/) | 네비게이션 레퍼런스 | 헤더, 사이드바, 메뉴 구조 참고 |
| [CTA Gallery](https://cta.gallery/) | CTA 섹션 | 랜딩/소개 페이지 전환 유도 참고 |

화면 설계 순서:

1. 사용자 플로우 먼저 작성
2. 화면 목록 작성
3. 각 화면의 주요 컴포넌트 정리
4. 빈 상태, 로딩, 에러, 성공 상태까지 추가
5. Figma로 와이어프레임 제작
6. 최종 디자인 시스템 적용

### 4단계: DB/API 설계

테이블, 관계, API 명세, 요청/응답 구조를 정리하는 단계입니다.

| 사이트 | 용도 | 팀에서 쓰는 방법 |
| --- | --- | --- |
| [ERDCloud](https://www.erdcloud.com/) | ERD 협업 | 팀 DB 설계, 테이블 관계 공유 |
| [dbdiagram.io](http://dbdiagram.io) | 빠른 ERD 작성 | 텍스트 기반으로 DB 구조 빠르게 작성 |
| [Postman](https://www.postman.com/) | API 테스트 | 요청/응답 확인, 팀 공유 |
| [Swagger](https://swagger.io/) | API 명세 | 백엔드 API 문서화 |
| [Hoppscotch](https://hoppscotch.io/) | 가벼운 API 테스트 | 빠른 요청 테스트 |
| [JSON Formatter](https://jsonformatter.org/) | JSON 정리 | 응답값 보기 좋게 확인 |
| [JWT.io](http://JWT.io) | JWT 확인 | 로그인 토큰 디코딩/검증 |

DB/API 설계 체크리스트:

- PK/FK/Unique/Null/Default가 명확한가?
- 상태값은 boolean, enum, code 중 무엇이 적절한가?
- 주문/결제/상태 변경처럼 이력이 필요한 데이터인가?
- 삭제는 실제 삭제인가, soft delete인가?
- API마다 요청값, 응답값, 에러 응답이 정리되어 있는가?
- 프론트가 화면을 그리는 데 필요한 값이 응답에 포함되어 있는가?

- 이해 안 된 코드는 팀 문서에 짧게 설명을 남긴다.

팀 코드 리뷰 체크리스트:

- 같은 코드가 반복되고 있지 않은가?
- 메서드 하나가 너무 많은 일을 하지 않는가?
- 변수명/함수명이 역할을 설명하는가?
- 예외 처리가 빠져 있지 않은가?
- 프론트와 백엔드의 데이터 이름이 일치하는가?
- 주석 없이도 읽히는 코드인가?

배포 전 체크리스트:

- 환경변수는 코드에 직접 적지 않았는가?
- DB 접속 정보가 공개 저장소에 올라가지 않았는가?
- 프론트 API 주소가 배포 환경에 맞는가?
- CORS 설정이 맞는가?
- 테스트 계정이 준비되어 있는가?
- 발표용 데모 시나리오가 준비되어 있는가?
