# 목적

이 페이지는 테스트 시나리오와 오류/버그를 데이터베이스로 관리합니다.

테스트는 단순 체크리스트가 아니라, 전제조건, 테스트 데이터, 수행 절차, 기대 결과, 실제 결과를 기록해야 합니다.

## 관리 방식

- 테스트 시나리오 DB: 기능별 테스트 케이스 관리
- 오류/버그 DB: 실패한 테스트나 발견된 문제 관리

## 테스트 상태값

예정 / 진행중 / 성공 / 실패 / 재테스트 / 보류

## 오류 상태값

등록 / 분석중 / 수정중 / 수정완료 / 재검증완료 / 보류

## 테스트 작성 기준

1. 테스트 ID를 부여합니다.
2. 테스트할 기능과 관련 API를 연결합니다.
3. 전제조건과 테스트 데이터를 작성합니다.
4. 수행 절차를 순서대로 작성합니다.
5. 기대 결과와 실제 결과를 비교합니다.
6. 실패 시 오류/버그 DB에 등록합니다.

## 테스트 범위 기준

### Week 7 스모크 QA (3일, 8/11~8/13)

- **압축 QA**: TC-001~008 핵심 경로만 3일 집중 수행 (고객 E2E + 관리자 상태 변경)
- 실패 건은 오류/버그 DB 등록 후 Week 8 통합·회귀에서 재검증
- DevCopilot·문서 추적성 점검은 Week 7 나머지 기간

### Week 5 MVP 테스트 (8/1 목표)

- API-001~API-008을 우선 검증하고, 옵션기능 범위까지 진행하는 경우 API-009 판매 항목 품절 상태 변경도 함께 검증합니다.
- 고객 흐름은 `홈·매장/포장(SCR-001) → 메뉴 조회 → 메뉴 상세 → 옵션 선택 → 기본 재료 빼기 → 장바구니·주문확인(SCR-005, 컨펌 팝업) → 주문 생성(API-005) → 가상 결제(SCR-007) → 주문 완료(SCR-008)`까지 확인합니다.
- 관리자 흐름은 `주문 목록 조회 → 주문 상세 확인 → 주문 상태 변경`까지 확인합니다.
- DB 기준으로 `orders`, `order_item`, `order_item_option`, `item_exclusion`, `payment`에 데이터가 올바르게 저장되는지 확인합니다.

### 옵션기능 테스트

- API-009 판매 항목 품절 상태 변경은 옵션기능으로 검증합니다.
- 메뉴 직접 품절, 핵심 재료 품절, 일반 기본 재료 품절, 베이스 재료 품절, 옵션 항목 품절 표시가 화면/API에서 같은 기준으로 동작하는지 확인합니다.

### 이후 확장 테스트

- API-010~API-020은 9주 로드맵의 후반 확장 테스트로 분리합니다.
- TC-015(영수증 출력)·TC-016(멤버십/쿠폰)은 Week 7~8 장치·멤버십 확장 테스트입니다.
- 관리자 메뉴 관리, 결제수단 설정, 장바구니 검증, 접근성 설정, 일별 매출 조회는 각 화면/API가 추가된 뒤 테스트 케이스를 작성합니다.
- 영수증 출력, QR/바코드 스캔, 장치/RTOS, 실제 PG 연동, 멤버십, 매출 통계는 확장 테스트로 분리합니다.

### 문서 추적성 테스트

- 기본기능 요구사항은 관련 시나리오, 화면, API, WBS 중 누락된 연결이 없는지 확인합니다.
- 화면 설계에 있는 기능은 API 명세 또는 WBS 작업 중 최소 1개 이상과 연결합니다.
- API 명세에 있는 요청/응답 필드는 DB 컬럼명 또는 계산 필드 기준과 어긋나지 않는지 확인합니다.
- 품절, 기본재료 제외, 추천 드레싱, 세트 선택처럼 문서 간 영향이 큰 기능은 요구사항 → 시나리오 → 화면 → API → DB → 테스트 순서로 함께 점검합니다.
