# 향후 확장 아이디어 — 샐러디 키오스크 프로젝트

> 이 문서는 요구사항조사서(통합본)에서 도출된 아이디어 중, **9주(7/2~9/2) 학원 팀 프로젝트 범위를 벗어나는 정식 서비스 수준의 기능**을 별도로 모아둔 것입니다.
MVP를 먼저 완성하고 여유가 생기면, 혹은 실제 서비스로 확장한다면 이 순서대로 검토하는 것을 추천합니다.
> 

---

## 1. 서브웨이 방식의 완전 자유 재료 조합

**아이디어**: 정해진 메뉴 없이 베이스·토핑·드레싱을 완전히 자유롭게 조합해서 나만의 샐러드를 만드는 기능.

**왜 지금은 보류하는가**
- 가격 계산 로직이 훨씬 복잡해짐 (조합마다 다른 가격 규칙 필요)
- 백엔드 데이터 모델이 “정해진 메뉴 + 옵션” 구조에서 “완전 커스텀 조합” 구조로 바뀌어야 해서 설계 난이도가 크게 상승
- UI도 옵션 선택형보다 훨씬 복잡한 흐름이 필요

**MVP 대안**: 정해진 메뉴 + 제한된 옵션(베이스 변경, 토핑 추가, 드레싱 변경) 조합으로 충분히 “커스텀 가능한 느낌”을 줄 수 있음.

**나중에 도전한다면**: MVP의 Option 테이블 구조를 확장해서, “메뉴 없이 옵션만으로 주문 생성”하는 별도 플로우를 추가하는 방식으로 접근.

---

## 2. 요일/시간대별 인기 토핑 통계 및 발주 예측

**아이디어**: 어떤 메뉴·토핑이 요일별/시간대별로 많이 팔리는지 시각화하고, 이를 기반으로 다음 날 발주량을 예측.

**왜 지금은 보류하는가**
- 의미 있는 통계를 내려면 최소 몇 주치 이상의 주문 데이터가 쌓여야 함 (9주 프로젝트 데모 시점엔 데이터 부족)
- 예측 로직(단순 평균 vs 머신러닝 등)까지 들어가면 프로젝트 범위를 크게 벗어남

**MVP 대안**: 관리자 화면에 “일별 매출”, “메뉴별 판매량” 정도의 단순 집계만 제공.

**나중에 도전한다면**: 더미 데이터로 몇 주치를 미리 채워넣고, 차트 라이브러리로 시각화하는 정도부터 시작.

---

## 3. 결제 승인-주문 저장 간 데이터 무결성 보장 (자동 망취소)

**아이디어**: 카드 결제는 승인됐지만 주문 데이터 저장에 실패한 경우, 자동으로 결제를 취소(망취소) 처리.

**왜 지금은 보류하는가**
- 실제 PG사(결제대행사) 연동이 필요한 영역이라, 학습 프로젝트에서 다루는 “결제 시뮬레이션” 수준을 넘어섬
- 망취소는 실무에서도 별도 트랜잭션 설계와 예외 처리가 필요한 고난이도 영역

**MVP 대안**: 결제 승인과 주문 저장을 하나의 트랜잭션으로 묶고, 실패 시 에러 로그를 남기고 사용자에게 재시도를 안내하는 정도로 단순화.

**나중에 도전한다면**: 실제 PG사 테스트 연동(예: 토스페이먼츠, 아임포트 등의 샌드박스 환경)을 붙이면서 함께 검토.

---

## 4. 웹 기반 CMS를 통한 실시간 장치 모니터링 및 원격 제어

**아이디어**: 여러 매장의 키오스크 하드웨어 상태(터치스크린, 프린터, 카드리더기)를 중앙에서 실시간으로 모니터링하고, 장애 시 원격으로 리부팅.

**왜 지금은 보류하는가**
- 여러 매장/여러 기기를 전제로 한 인프라라서, 4인 9주 프로젝트가 다루는 “단일 키오스크 시뮬레이션” 범위를 크게 벗어남
- 실제 하드웨어 장애 감지(용지 부족 센서 등)는 실물 장치 없이는 테스트 자체가 어려움

**MVP 대안**: 장치 상태값(정상/오류/네트워크끊김)을 DB에 기록해두고, 관리자 화면에서 조회만 가능하게 단순화.

**나중에 도전한다면**: 여러 키오스크 인스턴스를 시뮬레이션하는 데모 환경을 만들고, 상태 변경을 주기적으로 폴링하는 방식부터 시작.

---

## 5. 한국어/영어 언어 전환 UI

**아이디어**: 화면 상단에서 한국어/영어를 전환할 수 있는 다국어 지원.

**왜 지금은 보류하는가**
- 필수 기능은 아니고, 모든 텍스트 리소스를 분리 관리해야 해서 초기 설계에 다국어 구조를 넣지 않으면 나중에 추가하기 번거로움
- 9주 안에 핵심 주문 흐름을 완성하는 게 우선순위가 더 높음

**MVP 대안**: 한국어만 지원.

**나중에 도전한다면**: 초기 설계 단계부터 텍스트를 하드코딩하지 않고 i18n 라이브러리(react-i18next 등)로 분리해두면, 나중에 언어 추가가 수월함. 이 부분만은 미리 구조를 잡아두는 것을 권장.

---

## 우선순위 제안 (MVP 이후 도전한다면)

1. **5번(다국어)** — 구조만 미리 잡아두면 나중에 추가 부담이 적음
2. **1번(자유 재료 조합)** — 팀의 핵심 차별점이 될 수 있는 기능이라 여유가 되면 도전 가치 있음
3. **2번(발주 예측 통계)** — 데이터 시각화 연습에 좋음, 더미 데이터로 대체 가능
4. **3번(망취소)** — 실무 결제 시스템 이해에 좋지만 우선순위 낮음
5. **4번(CMS 원격 모니터링)** — 프로젝트 범위상 가장 나중, 정식 서비스화할 때 검토
