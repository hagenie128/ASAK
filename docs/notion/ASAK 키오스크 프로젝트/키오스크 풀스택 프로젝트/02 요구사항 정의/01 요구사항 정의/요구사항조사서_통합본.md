## 2. 인터뷰 목적

본 문서는 프로젝트 기획 전, 사용자의 요구와 운영상의 문제점을 파악하기 위해 작성한다.
실제 인터뷰가 어려운 관계로, 아래 세 가지 자료를 종합하여 가상의 고객·관리자·장치 담당자를 설정하고 요구사항을 도출하였다.

- 샐러디 실제 매장 키오스크(OKPOS 시스템) 사용 영상 관찰
- 샐러디 공식 자료(FAQ, 가맹 안내), 관련 뉴스 및 커뮤니티 피드백
- 팀원 각자의 실제 매장 이용 경험 및 인터뷰 시뮬레이션

## 4. 고객 인터뷰

### Q1. 키오스크를 사용할 때 가장 중요하다고 생각하는 점은 무엇인가요?

답변:
결제 과정이 단순해서 누구나 쉽게 쓸 수 있다는 점은 좋았지만, 샐러디는 베이스·토핑·드레싱까지 고를 게 많아서 화면이 복잡하게 느껴질 때가 있습니다. 뒤에 사람이 기다리고 있으면 눈치가 보여서 더 지치고요. 그래서 단계를 직관적으로 보여주거나, 매장 추천 조합을 기본값으로 주고 거기서 커스텀하는 흐름이 제일 중요한 것 같습니다. 메뉴를 빨리 찾고, 빠르게 결제까지 갈 수 있으면 좋겠습니다.

### Q2. 기존 주문/예약 과정에서 불편했던 점은 무엇인가요?

답변:
- 메뉴 사진이 없어서 어떤 메뉴인지 한눈에 확인하기 어려웠고, ‘우삼겹 웜볼’ 같은 이름만 봐서는 양이나 재료 구성을 가늠하기 어려웠습니다.
- 토핑/드레싱을 추가하는 과정이 불편했고, 지금 어떤 메뉴를 선택 중인지 확인하기 어려웠습니다.
- 이전 화면으로 돌아가기가 번거롭고, 실수로 취소를 누르면 담아둔 내용이 초기화된 적이 있습니다.
- 앱에서는 제공되는 칼로리·영양 정보가 키오스크에는 없어서 아쉬웠습니다.
- 특정 재료를 “빼주세요”라고 하고 싶은데(알레르기·기호), 추가(+) 옵션만 있고 제외(-) 옵션이 없어서 결국 카운터에 따로 말해야 했습니다.
- 드레싱 추천이 명확하지 않아 매번 검색해보거나 대충 골랐다가 입맛에 안 맞은 적이 많았습니다.
- 서브웨이처럼 재료를 자유롭게 조합하는 커스텀 기능이 없는 점도 아쉬웠습니다.
- 간편결제 선택 시 실제 지원되는 결제 수단이 화면에 안 보였고, 스탬프 적립을 결제 전/후로 두 번 물어봐서 번거로웠습니다.

### Q3. 키오스크 화면에서 꼭 보고 싶은 정보는 무엇인가요?

답변:
메뉴 사진, 가격, 칼로리, 주요 영양 정보(탄수화물·단백질·지방), 포함된 재료, 현재 선택 중인 메뉴/옵션, 총 결제 금액을 한눈에 보고 싶습니다. 옵션을 바꿀 때마다 칼로리·탄단지가 실시간으로 합산되어 보이면 가장 좋겠고, 비건 여부나 견과류·갑각류·우유 등 알레르기 유발 재료가 태그나 아이콘으로 표시되면 선택 단계에서 바로 확인할 수 있어 좋을 것 같습니다.

### Q4. 결제 또는 완료 후 어떤 정보가 제공되면 좋겠나요?

답변:
주문번호, 주문 내역, 예상 대기(조리)시간, 영수증 출력 여부를 확인하고 싶습니다. 스탬프가 정상 적립되었는지도 결제 완료 화면에서 바로 보이면 좋겠고, 가능하다면 종이 영수증 대신 모바일로 오늘 선택한 토핑 내역과 칼로리 요약을 받아볼 수 있으면 식단 기록에 도움이 될 것 같습니다.

## 5. 관리자 인터뷰

### Q1. 운영 중 가장 관리하기 어려운 부분은 무엇인가요?

답변:
재료가 매일 소포장으로 배송되어 그날 준비한 양만 판매하는 구조라서, 연어·아보카도·닭가슴살 같은 인기 토핑이 피크타임에 갑자기 떨어지는 경우가 많습니다. 이때 즉시 품절 처리하지 못하면 고객이 주문을 완료한 뒤에야 문제를 알게 되어 환불이나 메뉴 변경을 안내해야 하는 상황이 생깁니다. 옵션 하나만 품절되는 경우도 있고 카테고리 전체가 품절되는 경우도 있어 둘 다 관리할 수 있어야 합니다. 또한 손님마다 “이 재료 빼고 저 재료 추가” 같은 커스텀 요청이 다양해서, 주문서에 이 변경사항이 명확히 안 보이면 주방에서 오조리가 발생하기 쉽습니다.

### Q2. 관리자 화면에서 꼭 필요한 기능은 무엇인가요?

답변:
메뉴 등록/수정/삭제, 옵션 단위 품절 처리(터치 한 번으로 관련 메뉴 전체에 반영), 실시간 주문 목록 조회 및 상태 변경 기능이 필요합니다. 주문서나 주방 화면에는 기본 구성에서 “추가된 재료”와 “제외된 재료”를 색상이나 표시로 구분해주면 바쁜 시간대에도 헷갈리지 않을 것 같습니다.

### Q3. 주문/메뉴 상태는 어떻게 관리되어야 하나요?

답변:
주문 상태는 접수 → 준비중(조리중) → 완료 세 단계면 충분합니다. 조리랄 게 거의 없이 반가공 재료를 조합하는 방식이라 메뉴 하나 완성에 1분이 안 걸리거든요. 메뉴 상태는 판매중/품절/판매중지 정도로 구분하면 됩니다. 저희는 매장 식사와 포장 비율이 비슷해서, 주문이 들어올 때 “매장” 또는 “포장”이 명확히 구분되어 표시되면 실수를 줄일 수 있을 것 같습니다.

### Q4. 매출, 재고, 예약 현황 같은 정보가 필요한가요?

답변:
일별 매출, 인기 메뉴·토핑 판매량, 품절 메뉴 확인 정도면 충분합니다. 재고는 정확한 수량 추적보다는 “품절 여부(있음/없음)”로 단순하게 관리해도 괜찮고, 예약 기능은 저희 매장 특성상 크게 필요하지 않습니다. 요일·시간대별 인기 토핑 통계가 있으면 다음 날 발주량 조절에 참고가 될 것 같습니다.

## 6. 장치/시스템 담당자 인터뷰

### Q1. 키오스크에서 어떤 장치 이벤트가 필요하다고 생각하나요?

답변:
터치 입력, 카드 결제, 영수증 출력, 결제 오류 감지가 기본이고, 모바일 쿠폰이나 멤버십 QR/바코드를 스캔하는 빈도가 높으니 스캐너 상태를 실시간으로 체크하는 것도 필요합니다. 영수증 용지 부족이나 결제 후 카드 미수거 같은 상황을 미리 감지해서 안내하는 기능도 있으면 좋겠습니다.

### Q2. 결제 완료 후 어떤 처리가 필요하나요?

답변:
결제 승인 이후 주문 데이터(선택한 베이스·토핑·제외 재료 포함)가 유실 없이 저장되어야 하고, 동시에 관리자·주방 화면에 주문이 즉시 반영되어야 합니다. 주문번호도 함께 생성되어야 하고요. 만약 결제는 승인됐는데 주문 데이터 저장에 실패하는 경우를 대비해, MVP 단계에서는 최소한 에러 로그를 남기고 재시도하는 정도로 처리하고, 정식 서비스로 확장할 때 자동 취소(망취소) 로직을 검토하면 될 것 같습니다.

### Q3. QR/바코드/영수증 출력은 어떤 흐름으로 처리되어야 하나요?

답변:
결제 완료 → 주문번호 생성 → 주문 정보 저장 → 영수증 또는 화면 주문번호 표시 → 주문 완료 화면, 이 순서면 충분합니다. 모바일 쿠폰이나 멤버십 스캔은 결제 전에 먼저 처리되는 게 자연스럽고, 영수증은 선택적으로 출력하게 하는 게 좋습니다. 실물 프린터 연동이 부담되면 MVP에서는 화면 주문번호로 대체해도 됩니다.

### Q4. 장치 상태는 어떻게 확인하면 좋을까요?

답변:
카드 단말기 연결 상태, 프린터 용지/오류 여부, 네트워크 연결 상태를 관리자 화면에서 아이콘으로 볼 수 있으면 이상적입니다. 다만 학습 프로젝트 규모에서는 실시간 자동 감지보다, 상태값을 DB에 기록하고 관리자 화면에서 조회하는 정도로 단순화해도 충분합니다.

## 9. 벤치마킹 비교 분석

버거킹·공차·배스킨라빈스 관련 UX 자료를 조사한 결과, 평가가 엇갈렸다. 버거킹과 공차는 비교적 체계적으로 설계된 사례로 확인되었고, 배스킨라빈스는 실제 UX 디자이너 리뷰에서 부정적 평가가 다수 발견되었다.

| 브랜드 | 장점 (참고할 점) | 단점 (피해야 할 점) |
| --- | --- | --- |
| 버거킹 | 사진+텍스트 3x3 그리드, 결제=강조색·취소=중립색의 일관된 버튼 규칙, 재료 추가/삭제가 직관적 | 옵션이 많아지면 스크롤이 길어짐 |
| 공차 | 매장 직원 인터뷰 기반 워크숍으로 설계, **여러 옵션(음료·사이즈·토핑·당도·얼음)을 한 화면에서 모두 선택**하는 구조 채택, 멤버십 적립을 결제 직후 한 번에 처리 | 옵션 통합 화면이 첫 사용자에겐 정보량이 많아 보일 수 있음 |
| 배스킨라빈스 | - | “정보 나열식” 구성, 직원 주문보다 오래 걸림, 뒷사람 눈치로 인한 압박감 등 부정적 평가 다수 |

**우리 프로젝트 적용 방향**
- 공차 사례처럼 베이스/드레싱/토핑/스프를 한 화면에서 스크롤로 선택하는 현재 방식은 유지하되, 각 옵션 그룹에 짧은 설명을 추가 (요구사항 7번)
- 버거킹처럼 버튼 색상 규칙을 결제/확정=강조색, 취소/삭제=중립색으로 통일
- 멤버십 적립은 공차처럼 결제 완료 직후 1회만 처리 (요구사항 9번)
- 배스킨라빈스처럼 선택지가 지나치게 나열되지 않도록, 추천 조합을 기본값으로 먼저 보여주고 그 위에서 커스텀하는 흐름 채택

## 10. 결론

실제 샐러디 키오스크를 사용해보고 팀원 3인의 관점을 종합한 결과, **결제 과정 자체는 단순하고 주문 절차도 크게 복잡하지 않다는 공통된 평가**가 있었다. 다만 다음 지점들이 반복적으로 지적되었다.

- 메뉴 사진 부재로 인한 직관성 저하
- 옵션(토핑·드레싱) 추가 과정의 불편함과, 현재 선택 중인 메뉴 확인의 어려움
- 앱 대비 키오스크에서 칼로리·영양 정보가 부족한 점
- 재료를 자유롭게 빼는(-) 기능의 부재
- 간편결제 수단 미노출, 스탬프 적립의 이중 확인 절차

이를 바탕으로 **메뉴 이미지 제공, 실시간 영양정보 표시, 옵션 선택 UI 개선(설명 추가), 현재 선택 메뉴 확인 기능, 재료 제외 옵션, 결제수단 노출, 멤버십 적립 단순화**를 핵심 요구사항으로 도출하였다. 서브웨이식 완전 자유 조합이나 CMS 기반 원격 관제, 망취소 로직처럼 정식 서비스 수준의 기능은 8주 팀 프로젝트 범위를 벗어난다고 판단하여 별도 문서 [향후_확장_아이디어.md]로 분리하였다.
