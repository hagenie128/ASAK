# 00. 현재 운영 기준

> 기준일: 2026-07-13 · Notion에서 가져온 문서의 최신 판단 기준입니다.

## 팀 구성

현재 작업 팀은 다음 2인입니다.

| 팀원 | 주 담당 |
| --- | --- |
| 이하진 | 백엔드·DB, 메뉴/옵션·주문 API, 시드·통계 View |
| 김나연 | 키오스크·관리자 화면, 결제/관리자 연동, 문서·발표 |

과거 문서·회의록·워크로그에 남은 , 의 이름은 당시 기록이므로 이력 보존을 위해 유지합니다. 새 업무 배정과 현재 상태에는 사용하지 않습니다.

## DB 기준

현재 MySQL과 `asak-data/seed-v3/`는 짧은 명명 규칙을 사용합니다.

| 이전 명칭 | 현재 명칭 |
| --- | --- |
| `ingredient` | `ing` |
| `menu_ingredient` | `menu_ing` |
| `option_group` / `option_item` | `opt_group` / `opt_item` |
| `menu_option` | legacy 백업 전용 `menu_opt_legacy_20260710` |
| `option_policy` / `option_policy_item` | `opt_policy` / `opt_policy_item` |
| `menu_option_policy` | `menu_opt_policy` |
| `menu_nutrition` | `menu_nutr` |
| `payment_method_config` | `pay_method_cfg` |

새 API·화면·DB 작업은 이 문서와 [05 DB 설계](../../../../05%20DB%20설계%201d951ef04f0b83019b4281f04c7b12cc.md)를 기준으로 한다. 이전 명칭이 남은 API 명세·WBS 문서는 이관 전 기록으로 해석하며, 구현 시 현재 명칭으로 매핑한다.

## 시나리오·요구사항·화면·API ID 연계 기준

ID의 정본은 각 데이터베이스 CSV이며, 문서 간 참조는 아래 정식 ID만 사용한다.

| 구분 | 형식 | 정본 |
| --- | --- | --- |
| 요구사항 | `FWD-*`, `LMIS-*`, `KSD-*`, `RTOS-*` | 02 요구사항 정의 / 요구사항 목록 데이터베이스 |
| 시나리오 | `SC-###` | 03 사용자 시나리오 / 사용자 시나리오 데이터베이스 |
| 화면 | `SCR-###` | 04 화면 설계 / 화면 설계 데이터베이스 |
| API | `API-###` | 06 API 명세 / API 명세 데이터베이스 |
| 테스트 | `TC-###` | 09 테스트 오류 관리 / 테스트 시나리오 데이터베이스 |

### 정식 API ID

| API | 기능 | 대표 연계 화면 |
| --- | --- | --- |
| `API-001` | 카테고리 목록 조회 | `SCR-002` 메뉴 선택 |
| `API-002` | 메뉴 목록 조회 | `SCR-002` 메뉴 선택 |
| `API-003` | 메뉴 상세 조회 | `SCR-004` 메뉴 상세/옵션 |
| `API-004` | 메뉴 옵션 조회 | `SCR-004` 메뉴 상세/옵션 |
| `API-005` | 주문 생성 | `SCR-005` 장바구니/주문 확인 |
| `API-006` | 가상 결제 처리 | `SCR-007` 결제, `SCR-008` 주문 완료 |
| `API-007` | 관리자 주문 목록/상세 조회 | `SCR-009`, `SCR-010` |
| `API-008` | 관리자 주문 상태 변경 | `SCR-010` |
| `API-009` | 판매 항목 품절 상태 변경 | `SCR-011` |
| `API-010` | 관리자 판매 항목 목록 조회 | `SCR-011` |
| `API-011` | 관리자 메뉴 목록 조회 | 관리자 메뉴 관리 화면 |
| `API-012` | 관리자 메뉴 등록/수정 | `SCR-009` 관리자 메뉴 등록/수정 |
| `API-013` | 활성 결제수단 조회 | 관리자 결제수단 설정/결제 |
| `API-014` | 관리자 결제수단 설정 변경 | 관리자 결제수단 설정 |
| `API-015` | 관리자 일별 매출 조회 | 관리자 매출 요약 |
| `API-016` | 장바구니 검증 | 장바구니/주문 확인 |
| `API-017` | 접근성 설정 조회 | 접근성 설정 |
| `API-018` | 멤버십 스탬프 적립 | 결제 |
| `API-019` | 영수증 출력 요청 | `SCR-008` 주문 완료 |
| `API-020` | QR/바코드 스캔 처리 | `SCR-007` 결제 |

`API-HOLD-001`(관리자 로그인), `API-HOLD-002`(이전 영수증), `API-HOLD-003`(이전 QR/바코드), `API-HOLD-004`(장치 로그)는 보류/이력 항목이다. 새 시나리오·화면·요구사항에서는 `API-HOLD-*`를 연결하지 않고, 구현 범위에 포함된 영수증·QR은 각각 `API-019`, `API-020`을 사용한다.

### 작성 규칙

- 시나리오에는 `SC-### → 요구사항 ID → API-### → SCR-### → TC-###`를 모두 기입한다.
- 화면의 관련 API는 HTTP 메서드·경로와 함께 쓰되, ID는 위 정식 API ID를 사용한다.
- API의 관련 테이블에는 현재 DB명(`ing`, `opt_item`, `menu_ing`, `opt_policy` 등)을 사용한다. API 요청/응답의 외부 필드명(`ingredientId`, `optionItemId`)은 호환성을 위해 유지한다.
