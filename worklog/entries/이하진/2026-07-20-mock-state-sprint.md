## 2. 작업 목적

- 정적 UI 다음 단계인 mock 주문·결제·운영 상태 연결 작업을 실행 이슈와 구현 계획으로 고정한다.

## 3. 직접 구현 영역

- Kiosk에 loading, empty, error, modal 정적 컴포넌트와 결제·영수증·타임아웃 화면 상태 구조를 보강했다.
- Admin에 주문 관리용 정적 UI와 API mock 구조를 보강하고, 두 저장소의 구현 계획을 코드 실측 기준으로 갱신했다.
- Kiosk #5와 Admin #1을 생성해 WBS와 완료 조건을 연결했다.

## 4. 구현 로직 / 적용한 방식

- Kiosk는 수량 한도 토스트 → mock 결제 승인/실패 → 완료/오류 → 타임아웃 순서로 상태 전이를 연결한다.
- Admin은 repository/hook → adapter → Page 구조로 mock 데이터를 연결하고, Page의 직접 JSON import를 피한다.

## 5. AI 도움 영역

- 이번 문서도 결과만 적지 않고, **AI에게 어떤 기준으로 다시 시켰는지**를 남긴다.
- 첫 요청:
  - Codex에게 현재 구현 계획, 최근 커밋, 기존 이슈 목록을 함께 보고 "정적 UI 다음에 바로 실행할 수 있는 mock 상태 연결 작업"을 정리하라고 시켰다.
- 1차 초안에서 부족했던 점:
  - 화면 묶음 단위로 너무 넓게 잡혀 있어서 실제 다음 액션으로 옮기기 어려웠다.
  - 완료 조건이 "화면 구현" 수준에 머물러 있어 테스트 기준과 상태 전이 기준이 약했다.
  - Kiosk와 Admin 작업이 한 덩어리로 제안되어 저장소별 이슈 분리가 애매했다.
- 수정 지시:
  - "사용자 흐름 또는 운영 흐름 기준으로 다시 쪼개라"는 조건을 줬다.
  - Kiosk는 결제 승인/실패/타임아웃처럼 상태 전이 중심으로, Admin은 mock repository가 실제 페이지에 바인딩되는지 중심으로 재정리하게 했다.
  - 기존 열린 이슈/PR과 중복되지 않도록 번호 생성 전에 먼저 확인하게 했다.
  - 각 이슈에 완료 조건과 테스트 코드/시나리오 이름(TC-K01~K06, TC-A01~A06)을 붙이는 방향으로 다시 정리시켰다.
- 최종 채택 기준:
  - "오늘 바로 시작 가능한가"
  - "완료 여부를 화면/상태 변화로 검증할 수 있는가"
  - "저장소 책임이 분리되어 있는가"
- 그대로 쓰지 않은 부분:
  - 너무 큰 범위의 "mock 연결 전반" 같은 표현은 버렸다.
  - 정적 UI 완료를 기능 완료처럼 읽히게 만드는 표현은 제외했다.
- 최종적으로 사용한 부분:
  - 실행 이슈 후보 구조, 최근 커밋 기반 근거 묶기, 구현 계획과 워크로그 연결 흐름.
