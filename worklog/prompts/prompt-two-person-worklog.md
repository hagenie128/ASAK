# 통합 안내

이 파일은 이전 초안입니다. 별도 프롬프트와 스크립트를 쓰지 말고, [TEAM_WORKLOG.md](../TEAM_WORKLOG.md)만 사용하세요.

`TEAM_WORKLOG.md`에는 두 사람이 공통으로 쓰는 한 가지 프롬프트, 작성 기준, 이슈를 만드는 때가 함께 있습니다.

```text
ASAK 2인 팀 워크로그 기준으로 오늘 기록을 작성해줘.

작성자: {이하진 또는 김나연}
날짜: {YYYY-MM-DD}
오늘 메모:
- {한 일 / 변경한 파일 또는 화면 / 해결한 문제를 편하게 적기}
- {처음에 어떻게 판단했고, 진행 중 무엇을 확인·변경했는지}
- {AI에게 물어본 질문, 받은 답변 중 채택/수정/제외한 내용}
- {검증한 방법과 결과}
- {막힌 점, 다음에 개선할 점 또는 내일 할 일}

아래 규칙을 반드시 지켜줘.
1. 사실만 쓴다. 모르는 정보는 추측하지 말고 `[확인 필요]`로 표기한다.
2. 완료한 일은 과거형, 아직 할 일은 `TODO`로 구분한다.
3. `직접 구현`에는 내가 문제를 어떻게 판단하고 어떤 선택을 했는지, 내가 작성·수정한 내용을 쓴다.
4. `AI 도움`에는 아래 네 가지를 모두 구분해 쓴다: 내가 AI에 질문한 이유와 질문 / 받은 제안 / 채택하거나 수정한 내용과 이유 / 제외한 내용과 이유.
5. `개선 / 회고`에는 다음번에 더 빨리 또는 더 정확히 진행할 수 있는 한 가지를 쓴다. 단순히 "AI를 사용했다"고 쓰지 않는다.
6. `검증`에는 실행 명령어 또는 테스트 시나리오와 결과를 함께 쓴다. 검증하지 않았다면 `미검증`이라고 쓴다.
7. 한 작업이 끝났으면 이슈를 만들지 않는다. 다음 중 하나면 `이슈 초안 필요: 예`로 표시하고 초안을 만든다.
   - 아직 해야 할 작업이 남아 있음
   - 재현 가능한 버그가 있음
   - 다른 사람의 결정·확인이 있어야 진행 가능함
8. GitHub 이슈를 실제로 등록하지는 않는다. 초안만 작성한다.

출력 순서:
A. `오늘 요약` 표에 넣을 한 줄
B. 작업별 `오늘 작업` 미니 카드 (작업 목적 / 내 판단·진행 / 직접 구현 / AI 도움 / 이슈 / 검증 / 개선·회고 / 포트폴리오 요약)
C. `블로커 / 공유 사항`과 `내일 계획`
D. `이슈 초안 필요: 예|아니오`
E. 예인 경우에만 GitHub 이슈 초안: 제목 / 배경 / 할 일 / 완료 기준 / 권장 라벨

작업이 두 개 이상이면 작업별로 미니 카드와 이슈 판단을 나눈다.
```

## 두 사람이 같은 기준으로 쓰는 법

- 매일 각자 본인 파일 하나만 작성한다: `daily/{이름}/YYYY-MM-DD.md`
- 모든 작업은 **작업 목적 → 내 판단·진행 → 직접 구현 → AI 도움 → 검증 → 개선·회고** 순서로 쓴다.
- AI 도움은 "무엇을 받았나"보다 **왜 질문했고, 무엇을 판단해 채택·수정·제외했나**를 우선한다.
- 완료된 사실은 워크로그만 남긴다. 남은 작업·버그·의사결정 대기는 이슈 초안을 만든 뒤 팀이 확인하고 등록한다.
- 한 줄 요약에는 WBS, GitHub 이슈 번호, PR 링크가 이미 있을 때만 적는다. 번호를 지어내지 않는다.

## 빠른 입력 예시

```text
작성자: 김나연
날짜: 2026-07-18
오늘 메모:
- 관리자 주문 목록에서 상태 필터 UI를 만들고 API 파라미터와 연결했다.
- 처음에는 화면 상태별 조건문을 각각 만들려 했지만, 필터 값을 하나의 상태로 관리하는 편이 확장하기 쉽다고 판단했다.
- AI에게 "필터 상태와 API 파라미터를 분리하는 방법"을 물었다. 제안한 전역 상태는 현재 화면 규모에 과하다고 판단해 쓰지 않고, 필터 변환 함수만 수정해 적용했다.
- npm run build 성공, 상태별 필터 4개를 브라우저에서 확인했다.
- 취소 주문도 목록에 표시할지 팀 결정이 필요하다. 다음에는 API 응답 예시를 먼저 확인하고 UI를 구현한다.
```
