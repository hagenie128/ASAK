# 팀 워크로그 — 두 사람 공통 기준

매일 아래 **한 가지 프롬프트**만 Codex에 보내면 된다. 스크립트 실행이나 GitHub 이슈 등록은 필요 없다.

```text
오늘 워크로그 작성해줘. 내 이름은 {이하진 또는 김나연}이야.

오늘 메모:
- {오늘 한 일}
- {처음 판단한 방법과 실제 진행 과정}
- {AI에게 물어본 내용, 답변 중 채택·수정·제외한 이유}
- {검증한 방법과 결과}
- {막힌 점 / 다음에 개선할 점 / 내일 할 일}

ASAK/worklog/TEAM_WORKLOG.md 기준으로 아래를 작성해줘.
1. daily/{내 이름}/오늘날짜.md에 넣을 오늘 요약과 작업 카드
2. 작업별로 작업 목적, 내 판단·진행, 직접 구현, AI 도움, 검증, 개선·회고를 빠짐없이 작성
3. 사실만 쓰고, 모르는 정보는 [확인 필요]로 표시
4. 남은 작업·재현 가능한 버그·팀 결정 대기가 있을 때만 GitHub 이슈 초안을 마지막에 작성. 실제 등록은 하지 않음.
```

## 작업 카드에 꼭 들어갈 6가지

| 항목 | 무엇을 쓰나 |
|---|---|
| 작업 목적 | 무엇을 해결하거나 만들려고 했는지 |
| 내 판단·진행 | 처음 생각한 방법, 확인한 것, 중간에 바꾼 결정과 이유 |
| 직접 구현 | 본인이 작성·수정·결정한 코드·문서·디자인 |
| AI 도움 | 왜 질문했는지, 받은 제안, 채택·수정·제외한 이유 |
| 검증 | 명령어 또는 테스트 시나리오와 실제 결과. 안 했으면 `미검증` |
| 개선·회고 | 다음번에 더 낫게 할 한 가지 |

## 이슈를 쓰는 때

| 상황 | 워크로그 | GitHub 이슈 |
|---|---|---|
| 오늘 작업이 끝남 | 기록한다 | 만들지 않는다 |
| 다음에 해야 할 일이 남음 | 기록한다 | 초안을 만든다 |
| 재현 가능한 버그 | 기록한다 | 초안을 만든다 |
| 팀의 선택·확인이 필요함 | 기록한다 | 초안을 만든다 |

이슈는 항상 **초안만** 먼저 만든다. 두 사람이 확인한 뒤에만 GitHub에 등록한다.

## 작성 예시

```text
오늘 메모:
- 관리자 주문 목록의 상태 필터 UI를 만들고 API 파라미터와 연결했다.
- 처음에는 상태별 조건문을 따로 만들려 했지만, 필터 값을 하나의 상태로 관리하는 편이 확장하기 쉽다고 판단했다.
- AI에게 필터 상태와 API 파라미터를 분리하는 방법을 물었다. 전역 상태 제안은 현재 규모에 과해 제외하고, 변환 함수만 수정해 적용했다.
- npm run build 성공, 상태별 필터 4개를 브라우저에서 확인했다.
- 취소 주문 표시 여부는 팀 결정이 필요하다. 다음에는 API 응답 예시를 먼저 확인한다.
```

## 파일 위치

- 매일 기록: `worklog/daily/이하진/` 또는 `worklog/daily/김나연/`
- 기능별 상세 기록이 필요할 때만: `worklog/entries/{이름}/`
- 일일 파일이 없을 때만: `python worklog/scripts/init_daily.py --person {이름}`

평소에는 일일 파일을 열고 위 프롬프트의 결과를 붙여 넣으면 끝이다.
