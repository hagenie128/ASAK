### Admin 주문 관리 mock 바인딩 — SCR-010 · WBS2-036 · `feature`

- **작업 목적:** 하드코딩된 주문 목록을 mock repository와 hook으로 연결해 SCR-010 기본 흐름을 검증 가능하게 한다.
- **직접 구현:** `useOrdersQuery`가 `getAdminOrders` 전체 목록을 읽고 `usePagination`으로 slice 하도록 정리했다. `OrderManagementPreview`에 목록·상세 패널·페이지 전환·`orderLabels`·`formatCurrency`/`formatDateTime` 표시를 연결하고, `adminOrderApi` stub으로 환불·영수증 출력을 mock envelope 규칙에 맞게 처리했다.
- **AI 도움:** 먼저 Cursor에 `OrderManagementPreview`가 JSON을 직접 보지 않고 `repository → hook → page` 구조를 지키도록 연결 초안을 요청했다. 초안에는 목록 표시 자체는 빨리 잡혔지만, 페이지 전환 시 상세 패널 초기화와 mock envelope 필드(`orderStatus`, `paymentStatus`, `paymentMethod`) 표시 맥락이 약했다. 그래서 `useOrdersQuery`가 전체 목록을 읽고 `usePagination`이 slice 하도록 역할을 다시 분리해 달라고 지시했고, `orderLabels`·`formatCurrency`·`formatDateTime` 유틸을 통해 화면 표시 책임을 page에서 분리하도록 수정 요청했다. 이후 환불/영수증 액션도 "버튼만 만들지 말고 mock API stub의 성공/실패 조건을 toast에 연결하라"는 식으로 다시 피드백했고, 그 결과만 채택했다.
- **이슈:** 필터(주문/결제 상태·유형·날짜·검색)는 TODO로 남김 → WBS2-036 후속.
- **검증:** `npm run dev`로 주문 목록·상세·페이지 전환·환불 confirm/toast 흐름을 수동 확인 중이다.
- **포트폴리오 요약:** Admin 주문 화면을 repository 경계를 지키는 hook 패턴으로 mock 데이터에 연결하기 시작했다.
- **상세 기록 →** [2026-07-21 Admin mock 페이지 바인딩](../../entries/이하진/2026-07-21-admin-mock-page-binding.md)

### Admin 품절 draft·저장 mock — SCR-011 · WBS2-038 · `feature`

- **작업 목적:** 품절 화면의 양방향 이동·저장을 mock catalog와 동기화한다.
- **직접 구현:** `useSoldOutDraft`에 baseline·dirtyCount·선택 토글·`saveSoldOutCatalog` 저장 흐름을 구현했다. `SoldOutManagePage`는 카테고리 필터·패널별 `usePagination`·`AdminPagination`·저장 배지/버튼·접근성 checkbox 카드로 재구성했다. `adminMockRepository.saveSoldOutCatalog` stub을 추가했다.
- **AI 도움:** 처음에는 Cursor에게 품절 화면을 단순한 좌우 이동 UI로 붙여 보라고 시켰다. 그런데 첫 제안은 선택 상태와 저장 기준 상태가 섞여 있어 dirtyCount를 믿기 어려웠고, 저장 후에도 기준선이 즉시 갱신되지 않는 구조였다. 그래서 `available / soldOut / selectedAvailable / selectedSoldOut / baselineSoldOutKeys`를 역할별로 분리하고, "저장 전 임시 사본(draft)과 저장된 장부(baseline)를 구분해서 설명 가능한 코드로 다시 써 달라"고 요구했다. 또 패널별 페이지네이션과 전체 선택 UX를 추가로 요청했고, HMR 중 `onClick={}` 파싱 오류가 나온 뒤에는 핸들러 누락 지점을 같이 추적해 수정 방향을 되돌려 받았다. 최종적으로는 저장 후 baseline 재계산과 toast 흐름까지 확인된 버전만 반영했다.
- **이슈:** 저장 실패 fixture·롤백 시나리오는 아직 미구현.
- **검증:** dev 서버 HMR로 품절 이동·전체 선택·저장 toast·dirtyCount 초기화를 확인 중이다.
- **포트폴리오 요약:** 운영 화면에서 흔한 draft→저장 패턴을 mock repository와 분리된 hook으로 구현했다.
- **상세 기록 →** [2026-07-21 Admin mock 페이지 바인딩](../../entries/이하진/2026-07-21-admin-mock-page-binding.md)

### Admin 결제수단 draft·저장 mock — SCR-018 · WBS2-040 · `feature`

- **작업 목적:** 비활성화돼 있던 결제수단 화면을 mock repository와 draft hook으로 연결한다.
- **직접 구현:** `usePaymentMethodDraft`가 `getPaymentMethods`를 읽고 토글·저장 흐름을 처리하도록 보강했다. `PaymentMethodPage`에 loading·`AdminPaymentMethodRow`·`AdminSaveBar`·저장 toast를 연결했다.
- **AI 도움:** 결제수단 화면은 품절 화면에서 만든 draft 패턴을 그대로 복붙하지 말고, 토글 중심 화면에 맞게 더 얇게 가져가라고 Cursor에 지시했다. 초안에서는 저장 바 표시와 로딩 상태는 잡혔지만, 결제수단별 토글 변경과 저장 가능 상태 판단이 화면에서 읽기 쉽게 드러나지 않았다. 그래서 `getPaymentMethods` 조회, draft 변경 감지, `AdminSaveBar` 표시 조건을 분리해서 다시 정리하도록 피드백했고, "사용자 입장에서 지금 무엇이 바뀌었는지 바로 보이게 하라"는 기준으로 재수정했다. 현재 기록에는 그렇게 다듬은 흐름만 남긴다.
- **이슈:** 저장 실패 fixture·롤백 시나리오는 아직 미구현.
- **검증:** dev 서버에서 결제수단 토글·저장·toast 흐름을 확인 중이다.
- **포트폴리오 요약:** 운영 설정 화면도 동일한 draft→repository 저장 패턴으로 mock에 연결했다.
- **상세 기록 →** [2026-07-21 Admin mock 페이지 바인딩](../../entries/이하진/2026-07-21-admin-mock-page-binding.md)

### 공통 pagination·표시 유틸 — `refactor`

- **작업 목적:** 목록 화면마다 중복되던 slice·페이지 계산을 공통화한다.
- **직접 구현:** `usePagination`/`paginateItems` hook을 추가하고 `AdminPagination`을 sold-out·주문 목록에 재사용했다. `currency`·`date`·`confirm`·`toast` 유틸을 정리해 페이지에서 직접 포맷하지 않도록 했다.
- **AI 도움:** 이 부분은 단순 생성보다 리뷰 보조에 더 많이 썼다. Cursor가 여러 페이지에서 반복되는 slice 계산과 포맷 로직 위치를 먼저 훑고 후보를 제안했고, 나는 그중에서 실제로 재사용 가치가 있는 것만 추렸다. 특히 "페이지 컴포넌트에서 직접 숫자/날짜 포맷하지 말고 유틸로 밀어내자", "pagination은 화면별 구현이 아니라 배열 기준 공통 로직이어야 한다"는 식으로 방향을 여러 번 좁혀 가며 수정 지시를 줬다.
- **이슈:** -
- **검증:** 주문·품절 화면에서 페이지 경계·항목 수 변경 시 page 보정 동작을 확인했다.
- **포트폴리오 요약:** mock 목록 UI의 공통 pagination 레이어를 추출해 후속 화면 연결 비용을 줄였다.
